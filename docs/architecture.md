# Clawbot Architecture

## Purpose

This document describes the current production architecture for the Hetzner-hosted
OpenClaw deployment in this repo. It reflects what is actually running now, not the
older intended design.

It also records the main architectural rough edges so future cleanup work has a
clear starting point.

## Current deployment shape

Production stack:

- `live/prod/fsn1/clawbot`

Infrastructure:

- Hetzner Cloud VM
- Ubuntu 24.04
- server type `cpx22`
- persistent `/opt` volume attached separately from the root disk

Core services:

- OpenClaw gateway in rootless Podman under the `openclaw` user
- `nginx` for public HTTPS ingress
- `certbot` for Let's Encrypt certificate management
- local FastAPI Telegram relay service at `127.0.0.1:9000`
- five private same-host bot runtimes under the `openclaw` user:
  - `clawbot-bob-runtime`
  - `clawbot-stacks-runtime`
  - `clawbot-jennifer-runtime`
  - `clawbot-steve-runtime`
  - `clawbot-number5-runtime`

Public ingress:

- `https://agents.satoshis-plebs.com/`
- root path intentionally returns `404`
- only Telegram webhook paths are intended for public use

## Bootstrap and rebuild model

Bootstrap is driven by:

- `modules/clawbot_server/bootstrap-node-runner.sh`

The node fetches the bootstrap runner from GitHub `main` via:

- `OPENCLAW_BOOTSTRAP_RUNNER_URL`

Normal rebuild flow:

- replace `hcloud_server.clawbot`
- preserve the `/opt` volume
- reattach `/opt` to the replacement node
- rerun bootstrap

Operational rule:

- do not test a rebuilt node until `/var/log/cloud-init-output.log` ends with:
  - `openclaw node bootstrap complete.`
  - `Cloud-init ... finished`

## Durable vs ephemeral state

Durable state lives under `/opt/clawbot`.

Important durable paths:

- `/opt/clawbot/config/openclaw.json`
- `/opt/clawbot/config/runtime/llm.yaml`
- `/opt/clawbot/config/secrets/llm.env`
- `/opt/clawbot/config/secrets/telegram.env`
- `/opt/clawbot/config/agent-config/`
- `/opt/clawbot/config/telegram-webhook/`
- `/opt/clawbot/tls/letsencrypt/`
- `/opt/clawbot-root/bootstrap/openclaw-node-bootstrap-runner.sh`
- `/opt/clawbot-root/secrets/`
- `/opt/clawbot/state/`
- `/opt/clawbot/work/`
- `/opt/clawbot/logs/`

Ephemeral state:

- `/srv/openclaw`
- the root disk generally

Rule of thumb:

- if a file is needed after a rebuild, it should live under `/opt/clawbot` or be
  regenerated by bootstrap
- root-only recovered bootstrap artifacts and agent-specific high-value secrets live under
  `/opt/clawbot-root`

## Secret boundaries

Current secret layers:

- shared gateway env:
  - `/opt/clawbot/config/.env`
  - `/opt/clawbot/config/secrets/llm.env`
  - `/opt/clawbot/config/secrets/telegram.env`
- agent-specific root-owned secret scaffolding:
  - `/opt/clawbot-root/secrets/<agent-id>.json`
  - OpenClaw `secrets.providers.agent_<agent-id>_root`

The `agent_<agent-id>_root` providers exist to keep future per-agent credentials, such as
Nostr private keys or treasury-related material, out of shared env and out of
`openclaw.json`. They use a root-run exec provider and a narrow sudoers rule so the secret
stays root-owned at rest.

Security limitation:

- This is storage separation, not hard process isolation.
- All bots still share one host and one OS user.
- A truly high-assurance Stacks or Treasurer secret boundary still requires a separate
  runtime boundary on a separate host or VM.

## Runtime request path

### Control UI

- OpenClaw gateway listens on `127.0.0.1:18789`
- operator typically reaches it through SSH local forwarding:
  - `ssh -N -L 18789:127.0.0.1:18789 mcintosh@<host>`

### Telegram ingress

Telegram webhook flow:

1. Telegram sends `POST` to `https://agents.satoshis-plebs.com/telegram/<bot>`
2. `nginx` forwards to the local relay at `127.0.0.1:9000`
3. the relay validates `X-Telegram-Bot-Api-Secret-Token`
4. the relay forwards to the per-bot private runtime
5. the runtime returns structured actions
6. the relay performs outbound Telegram sends with the correct bot token

Current private runtime listener ports:

- `bob` -> `18920`
- `stacks` -> `18921`
- `jennifer` -> `18922`
- `steve` -> `18923`
- `number5` -> `18924`

Legacy OpenClaw Telegram listener ports `18890`-`18894` still exist during the migration,
but they are no longer the primary ingress path for public Telegram traffic.

## OpenClaw configuration model

The effective generated runtime config is:

- `/opt/clawbot/config/openclaw.json`

Key current settings:

- `session.dmScope = "per-account-channel-peer"`
- Telegram owner access uses `allowlist`
- owner Telegram user id allowlisted: `1619231777`
- `agents.defaults.model.primary = "openrouter/auto"`

Current private runtime mapping:

- `bob` -> runtime `bob-runtime` -> agent id `orchestrator`
- `stacks` -> local runtime container `clawbot-stacks-runtime` -> agent id `podcast_media`
- `jennifer` -> local runtime container `clawbot-jennifer-runtime` -> agent id `research`
- `steve` -> local runtime container `clawbot-steve-runtime` -> agent id `engineering`
- `number5` -> local runtime container `clawbot-number5-runtime` -> agent id `business`

Current dashboard identity mapping:

- internal id `orchestrator` displays as `Bob`
- internal id `podcast_media` displays as `Stacks`
- internal id `research` displays as `Jennifer`
- internal id `engineering` displays as `Steve`
- internal id `business` displays as `Number 5`

Current agent binding mechanism:

- explicit `agents.list`
- explicit `bindings` by Telegram `accountId`

This is important: without those explicit bindings, OpenClaw collapsed back to a
single default `main` agent.

## Agent identity and prompt files

Human-maintained role/prompt files live here:

- `/opt/clawbot/config/agent-config/agent-fleet.yaml`
- `/opt/clawbot/config/agent-config/orchestrator/policy.md`
- `/opt/clawbot/config/agent-config/specialists/business.md`
- `/opt/clawbot/config/agent-config/specialists/engineering.md`
- `/opt/clawbot/config/agent-config/specialists/research.md`
- `/opt/clawbot/config/agent-config/specialists/podcast_media.md`

Alias-name files also exist:

- `specialists/jennifer.md`
- `specialists/stacks.md`
- `specialists/steve.md`

What these files do today:

- `agent-fleet.yaml` is the durable role map
- the markdown files act as persona / policy prompts

What they do not do today:

- they are not the sole source of truth for Telegram routing
- they are not a complete first-class OpenClaw native multi-agent package by themselves

## LLM and provider model

LLM runtime files:

- `/opt/clawbot/config/runtime/llm.yaml`
- `/opt/clawbot/config/secrets/llm.env`

Current provider design:

- provider type: OpenAI-compatible
- base URL: OpenRouter
- default runtime model path in generated OpenClaw config: `openrouter/auto`

This was necessary because routed Telegram agents were otherwise defaulting to:

- `anthropic/claude-opus-4-6`

and failing before reply due to missing Anthropic auth.

## OpenClaw state model

OpenClaw runtime state is stored under:

- `/opt/clawbot/state/.openclaw/agents/`

Typical contents:

- `agent/models.json`
- `sessions/sessions.json`
- session transcript files under `sessions/*.jsonl`

This is runtime state, not the human-authored role definition.

## Security posture

Current intended operator model:

- one trusted operator
- Telegram DMs from the allowlisted owner
- Control UI accessed over SSH tunnel, not exposed publicly

Current public exposure:

- `80/tcp`
- `443/tcp`

Current public application surface:

- Telegram webhook endpoints only

## Current architectural rough edges

The system works, but it is still a pragmatic integration rather than a clean
OpenClaw-native architecture.

Main rough edges:

1. Agent identity is split across multiple places
   - `agent-fleet.yaml`
   - specialist markdown files
   - generated `openclaw.json`
   - dashboard display names now come from `agents.list[].identity.name`

2. Telegram routing is generated separately from the durable role map
   - the mapping is correct now
   - but it is not yet sourced from one canonical definition

3. Legacy `main` agent state still exists
   - it reflects the earlier single-agent phase
   - it is confusing when inspecting runtime state

4. Prompt/config conventions are repo-defined
   - effective and understandable
   - but not obviously aligned with a canonical OpenClaw agent packaging standard

## Recommended cleanup plan

### Phase 1: stabilize current production shape

- keep the current Telegram routing and OpenRouter defaults
- prove rebuilds stay green
- avoid structural refactors until the automation is boring

### Phase 2: establish a single source of truth

- choose one canonical agent definition source
- likely candidates:
  - `agent-fleet.yaml`
  - a more OpenClaw-native agent declaration format
- generate Telegram account bindings from that source rather than duplicating them

### Phase 3: remove legacy state assumptions

- decide how to retire or migrate the old `main` agent state
- make inspection output less confusing for future operators

### Phase 4: normalize agent packaging

- define a clearer standard for:
  - persona
  - routing
  - model defaults
  - memory/runtime state
  - tool/permission boundaries

- if desired, move toward a more explicitly OpenClaw-native multi-agent layout

### Phase 5: add new specialists cleanly

When adding a future bot such as a treasurer bot:

1. define role/prompt file
2. add durable role-map entry
3. add Telegram account id and webhook path
4. add explicit agent binding
5. verify OpenRouter default applies
6. rebuild and test after bootstrap completion

## Practical takeaway

The current system is best understood as:

- a Hetzner VM
- with durable `/opt` state
- running OpenClaw in rootless Podman
- fronted by nginx/certbot
- with a local Telegram relay
- and explicit generated multi-agent Telegram bindings

It is operational now, but still a good candidate for later cleanup into a more
standardized agent architecture.
